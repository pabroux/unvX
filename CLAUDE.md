# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

unvX is a standalone Home Manager configuration using Nix flakes to manage dotfiles and software installation on macOS. It uses `mkOutOfStoreSymlink` for dotfile management — config edits take effect immediately without `home-manager switch`.

## Commands

**Apply configuration:**
```bash
home-manager switch
```

**First-time installation:**
```bash
./install                           # Setup git hooks, generate unvX.nix, link flakes
nix run home-manager -- switch      # Initial Home Manager installation
```

**Development shell (provides formatters and prek):**
```bash
nix develop
```

**Run prek checks:**
```bash
nix develop . -c prek run -a
```

**Formatters:**
- Nix: `alejandra`
- Shell: `shfmt`
- Lua: `stylua`
- YAML: `yamlfmt`

## Architecture

### Entrypoint Flow
```
flake.nix → reads unvX.nix → host/<hostname>/user/<user>/home.nix → imports module/*/default.nix
```

`unvX.nix` is generated by `./install` and contains absolute paths (`directory.root`, `directory.module`, `directory.host`). It is passed to every module via `extraSpecialArgs = {inherit unvX;}` in `flake.nix`.

### Key Directories

- **`flake.nix`**: Inputs (nixpkgs, home-manager, flake-utils), outputs (home configurations), and dev shell
- **`host/`**: Home Manager profiles matched by `$USER@$(hostname)`, falling back to `$USER`
- **`module/`**: Self-contained modules for each software
- **`unvX.nix`**: Generated file — never edit manually

### Module Pattern

Every module is `module/<name>/default.nix` taking `{config, pkgs, lib, unvX, ...}`. Three building blocks, used alone or combined:

**1. Package declaration:**
```nix
home.packages = [ pkgs.<name> ];
```

**2. Config symlinking** (the core pattern — edits apply without rebuild):
```nix
home.file."<label>" = {
  source = config.lib.file.mkOutOfStoreSymlink "${unvX.directory.module}/<module>/<config>";
  target = ".config/<app>/<config>";
};
```
The attribute name (`<label>`) is just a Home Manager key. The `target` determines actual placement relative to `$HOME`.

**3. Activation hooks** (post-install scripts):
```nix
home.activation.<hookName> = lib.hm.dag.entryAfter ["installPackages"] ''...''
```

### Adding a Module

1. Create `module/<name>/default.nix`
2. Add `"<name>"` to the `modules` list in `host/.../home.nix` (alphabetically)

The `modules` list is also reused by `allowUnfreePredicate` — adding a module name automatically allows its unfree packages.

### Dynamic Loading via `builtins.readDir`

Three modules auto-wire files from directories — drop a file in the right place and it's picked up without editing `default.nix`:

- **neovim**: Every `.lua` in `module/neovim/lua/plugins/` is symlinked to `~/.config/nvim/lua/plugins/`
- **claude-code**: All files in `agents/` and nested `skills/<dir>/` are symlinked to `~/.config/claude/`
- **homebrew**: Only imports `app/<cask>/default.nix` for casks that have a config subdirectory (`builtins.filter ... builtins.pathExists`)

### Git Hooks

The `install` script generates `.git/hooks/{pre-commit,commit-msg}` using a **nix shebang** (`#! nix develop . --command bash`) that bootstraps into the dev shell and calls `prek hook-impl`. Hook definitions live in `prek.toml` — all use `language = "system"` (rely on dev shell tools).

### Two `settings.json` Scopes

- **`.claude/settings.json`** (project-level): PostToolUse hooks that auto-format on Write/Edit during development
- **`module/claude-code/settings.json`** (user-level): Deployed to `~/.config/claude/settings.json` — permissions, plugins, global preferences

### Hybrid Environment Strategy

- **Nix (unvX)**: Foundation tools (shell, editor, git, tmux)
- **Mise**: Per-directory runtime versions (Node, Python, Go)
- **Homebrew**: macOS GUI applications (managed via homebrew module)
